# ZenTao Auth And Precheck

所有九大核心能力，在真正执行前都先经过这一层。

## 1. 鉴权检查

先确认本地已具备：

- REST token
- MVC `zentaosid`
- 必要时可重登的本地账号信息

默认行为：

- 平时优先直接使用本地已登录的 `token + sid`
- 只有当 REST `401/403` 或 MVC 跳登录页时，才触发一次静默重登
- 静默重登依赖本地保存的账号密码，因此不能假设只有登录命令才会读取这份本地配置
- 静默重登成功后，应同步更新本地配置中的最新 `token + sid`
- 该规则对个人视图、管理视角、晨会、周报、负荷检查、停滞排查等命令统一适用

如果出现下面任一现象，优先判定为登录态失效：

- REST 返回 `401`
- MVC 页面跳到登录页
- MVC 返回脚本重定向到 `user-login-*`

处理规则：

1. 若本地已有可用登录信息，先触发一次重新登录，再重试原动作。
2. 若没有可用登录信息，再要求用户显式登录。
3. 不要在鉴权失败状态下继续猜测业务问题。

## 2. 写操作前置校验

在创建/修改前，先确认：

- 对象 ID 是否已知
- 若未知，能否通过名称检索或链接解析得到
- 若关联需求，需求 ID/链接是否已知
- 项目与执行是否已知
- 若是团队任务，成员列表是否已归一化

缺关键信息时，不要直接提交写请求。

## 3. 项目与执行前置校验

涉及创建任务时：

- 普通任务：先确认执行是否明确
- 需求拆分任务：先确认需求归属项目，再确认执行是否属于该项目
- 如果要自动创建当月执行，必须沿用上个执行的团队/权限配置，避免成员丢权限

## 4. 写后回读校验

写操作成功后，不要只信接口返回“成功”两个字，要回读关键结果：

- 任务创建：回看任务 ID、执行、模式、优先级、参与人、需求关联
- 指派变更：回看新负责人或团队成员是否真的变化
- 状态流转：回看状态是否真正更新
- 报工：回看耗时与备注是否真正落库

## 5. 查询前置校验

查询前先判断：

- 这是个人视角还是管理视角
- 是否是专题统计，而不是普通列表查询
- 是否需要先解析链接
- 是否需要先展开团队缓存

不要在没分清视角的情况下直接调用查询能力。
